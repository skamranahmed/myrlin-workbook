# Phase 2 — Feature Inventory

> **Scope:** This is a product-oriented inventory of the reference repository. It describes user-facing capabilities, their purpose, workflows, dependencies, complexity, usage frequency, and product importance. It does not prescribe an implementation or recommend changes.
>
> **Classification:**
> - **Core:** central to the product's primary value proposition.
> - **Important:** materially supports the primary workflows but is not the defining center of the product.
> - **Optional:** useful capability that serves a subset of users or a secondary workflow.
> - **Nice to Have:** convenience, polish, or specialized enhancement.
> - **Legacy:** archived, superseded, de-emphasized, or retained mainly for historical/backward compatibility.
>
> **Complexity scale:** Low = focused interaction or configuration; Medium = multi-step stateful workflow; High = several domains, failure modes, or external dependencies; Very High = live processes, isolation, streaming, identity, or multiple external systems.
>
> **Frequency scale:** Continuous = active/background operation; Daily = common during normal work; Weekly = used during planning/review cycles; Occasional = used when a particular situation occurs; Rare = setup, recovery, or specialized operation.

## Feature Overlap and Boundaries

### Project organization versus task orchestration

- **A1 hierarchy** answers *where does this work belong?* It organizes projects, focuses, and sessions.
- **D3 feature board** answers *what project work is planned and what phase is it in?*
- **E3 orchestration board** answers *which executable agent tasks are queued, running, blocked, under review, or complete?*
- **E1 worktree tasks** are executable isolated units; they are not simply another name for a project feature.

The repository describes both a four-column feature board and a five-column worktree-task board. They appear conceptually distinct, but the exact product boundary remains unresolved.

### Session execution versus session observation

- **B1 embedded terminal** is interactive and can resume/control a session.
- **B6 live mirror** is read-only and observes a session without taking control.
- **B4 activity detection** classifies activity for both surfaces but does not itself provide terminal access.

### Search versus semantic discovery

- **A8 global search** uses names, paths, tags, and transcript content for deterministic navigation.
- AI-assisted insight/finder capabilities use model interpretation to find or summarize related material. They depend on optional AI access and are not substitutes for basic search.

### Cost versus quota/usage

- **F1 cost tracking** estimates historical spend and token consumption.
- **F2 usage meter** shows current account/model windows and reset timing.
- **L1 archived quota widget** was the older quota-oriented surface and is not current.

### Local versus remote notifications

- **B5 desktop notifications** alert a user near the host.
- **H6 mobile push** alerts a paired user away from the host.
- Both rely on activity/task events, but their delivery, permission, and failure behavior differ.

### Local access versus remote deployment

- **G1 local authentication** protects the Workbook service.
- **G4 LAN access** makes it reachable on a local network.
- **G5 quick tunnels** provide temporary remote reachability.
- **G6 named tunnels/Cloudflare Access** provide stable or externally gated reachability.
- **G7 always-on supervision** keeps the host available; it does not itself provide authentication.

### Context transfer versus task spinoff

- **D5 context export/refocus** creates a reusable handoff or fresh context for one session.
- **E11 task spinoff** turns extracted work into multiple structured, isolated tasks. It is a specialized orchestration workflow built on the more general context-transfer idea.

### Backup and recovery boundaries

- **A10/A11** preserve and recover Workbook/session operating state.
- **J5** protects or restores frontend assets.
- Neither is clearly documented as a general export/import or multi-machine backup system; portability remains an unknown from Phase 1.

## Inventory Notes and Status Caveats

- The repository contains material from several product generations. `package.json`/`CHANGELOG.md` describe a newer 1.2 line, while portions of `README.md` still describe multi-provider support as alpha.
- “Current” means the repository contains user-facing evidence for the capability; it does not guarantee that every documented edge case is stable.
- The status of **E11 Task Spinoff**, native mobile parity, and some mobile offline behavior is unclear because documentation and implemented surface evidence do not align.
- The inventory intentionally excludes purely internal quality mechanisms, such as provider-name grep tests, as product features.
- The inventory also does not treat a general user-data export/backup feature as present because Phase 1 identified it as an unknown rather than a documented capability.
